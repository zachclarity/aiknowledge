What is RAG?

RAG (Retrieval-Augmented Generation) is a technique that enhances Large Language Models by giving them access to external knowledge. Instead of relying solely on what the model learned during training, RAG:

  1. 1 Retrieves relevant information from a knowledge base (your codebase)
  2. 2 Augments the user's question with that context
  3. 3 Generates an accurate response using the LLM

💡 Why RAG for Code?

LLMs don't know about your specific codebase. RAG bridges this gap by letting you ask questions like "What S3 bucket is defined?" or "What happens in the test stage?" and getting answers based on YOUR actual code.

Docker Setup Explained

The docker-compose.yml defines three interconnected services:

docker-compose.yml 
services:
  # Service 1: Ollama - Local LLM Server
  ollama:
    image: ollama/ollama:latest
    container_name: ollama
    ports:
      - "11434:11434"  # Ollama API port
    volumes:
      - ollama_storage:/root/.ollama  # Persist downloaded models
      - ./Modelfile:/root/Modelfile
    entrypoint: >
      /bin/sh -c "
      ollama serve & 
      sleep 5;
      ollama pull qwen2.5-coder:0.5b;  # Download models on startup
      ollama create fastapi-gen -f /root/Modelfile;
      wait"

  # Service 2: ChromaDB - Vector Database
  chromadb:
    image: chromadb/chroma:latest
    container_name: chromadb
    ports:
      - "8000:8000"  # ChromaDB API port
    volumes:
      - chroma_data:/chroma/chroma  # Persist vector data
    environment:
      - IS_PERSISTENT=TRUE  # Enable persistence

  # Service 3: Backend - Your Python App
  backend:
    build: ./onedemo
    container_name: one_app
    ports:
      - "8787:8080"
    environment:
      - CHROMA_HOST=chromadb  # Connect to ChromaDB container
      - CHROMA_PORT=8000
      - OLLAMA_HOST=http://ollama:11434  # Connect to Ollama
    depends_on:
      - ollama
      - chromadb  # Wait for dependencies

volumes:  # Named volumes for data persistence
  ollama_storage:
  chroma_data:

🔌 Ports Exposed

  • 11434 - Ollama API
  • 8000 - ChromaDB API
  • 8787 - Your App

💾 Data Volumes

  • ollama_storage - Models
  • chroma_data - Vectors

🔗 Service Network

Containers reference each other by name (e.g., chromadb)

def index_code(path_to_index):
    # Directories to skip
    ignored_dirs = {'.git', 'venv', '__pycache__', 'node_modules'}
    
    # Supported file types
    supported_ext = {".py", ".java", ".js", ".tf", ".yml", ".yaml"}
    
    # Walk through files...
    for root, dirs, files in os.walk(path_to_index):
        for file in files:
            # Read file content
            with open(file_path) as f:
                content = f.read()
            
            # Split into chunks
            chunks = get_chunks(content, CHUNK_SIZE)
            
            for chunk in chunks:
                # Get embedding from Ollama
                response = ollama.embeddings(model=EMBED_MODEL, prompt=chunk)
                
                # Store in ChromaDB
                collection.add(
                    ids=[chunk_id],
                    embeddings=[response["embedding"]],
                    documents=[chunk],
                    metadatas=[{"source": file_path}]
                )

Key Insight: Each code file is split into ~1500 character chunks. Each chunk is converted to a vector (embedding) and stored with metadata about its source file.

🔍 Query Function

query_code() - Search and Generate 
def query_code(user_query):
    # Step 1: Embed the user's question
    query_embed = ollama.embeddings(
        model=EMBED_MODEL, 
        prompt=user_query
    )["embedding"]
    
    # Step 2: Find similar code chunks (top 3)
    results = collection.query(
        query_embeddings=[query_embed], 
        n_results=3
    )
    
    # Step 3: Build context from results
    context = "\n---\n".join(results["documents"][0])
    
    # Step 4: Generate answer using LLM
    prompt = f"[INST] Use the code snippets below to answer: {user_query}\n\nContext:\n{context} [/INST]"
    
    response = ollama.generate(model=LLM_MODEL, prompt=prompt)
    return response["response"]

Dependencies Explained

The requirements.txt lists the Python packages needed:

1
chromadb

The Python client for ChromaDB. Provides methods to create collections, add embeddings, and query for similar vectors.

Used for: client.get_or_create_collection(), collection.add(), collection.query()
2
ollama

Official Python library for Ollama. Enables embedding generation and LLM inference through simple function calls.

Used for: ollama.embeddings(), ollama.generate()
3
requests

Classic HTTP library for Python. A dependency used internally by other packages for making API calls.

4
httpx & httpcore

Modern async-capable HTTP client. Used by the Ollama library for communicating with the Ollama server, supporting both sync and async operations.

requirements.txt 
chromadb      # Vector database client
requests      # HTTP library (dependency)
ollama        # Ollama Python SDK
httpx         # Async HTTP client
httpcore      # HTTP transport library

Running the Demo

1

Start the Services

docker-compose up -d

2

Processes all supported files and stores embeddings in ChromaDB.

python one.py

Runs example queries. Modify the script to add your own questions!
